# .NET Core Library Migration Plan

## Overview

This plan outlines the migration of the TypeScript `@subscrio/core` library to a .NET equivalent. The .NET version will maintain the same architecture, domain model, and public API patterns while leveraging .NET-specific features and conventions.

## ⚠️ CRITICAL: Functional Equivalence with C# Idioms Philosophy

**THIS IS THE ABSOLUTE MOST IMPORTANT REQUIREMENT FOR THIS PROJECT.**

### Core Principle

**Every single method MUST be analyzed from the TypeScript source code to understand ALL functionality and logic, then implemented using appropriate C# idioms and best practices. The goal is functional equivalence, not literal syntax translation.**

### Mandatory Process for Each Method

1. **Understand the TypeScript Functionality**

   - Open the exact TypeScript file containing the method
   - Read and understand the complete method implementation
   - Identify all operations: validation, repository calls, business logic, error handling
   - Understand the purpose and behavior of each operation
   - Note all edge cases, error paths, and business rules

2. **Map Functionality to C# Equivalents**

   - Understand what each TypeScript operation does, not how it's written
   - Use appropriate C# patterns and idioms for each operation
   - Validation → FluentValidation (using correct FluentValidation API, not Zod patterns)
   - Repository calls → Repository interface calls (same pattern)
   - Error handling → Exception throwing (same pattern)
   - Data transformations → Mapper calls (same pattern)
   - Business rules → Domain entity methods or service calls (same pattern)
   - Property mutations → Direct property assignment (C# classes use setters, not `with` expressions)

3. **Verify Complete Functional Coverage**

   - All TypeScript functionality must be present in C# code
   - All validation rules must match (same checks, same error messages)
   - All business rules must match (same domain logic)
   - All error paths must be preserved
   - All edge cases must be handled identically
   - Behavior must be functionally equivalent in all cases

4. **Test-Driven Verification**

   - Every TypeScript test must have a corresponding .NET test
   - Tests must verify the same behavior
   - Edge cases must be tested identically

### What This Means in Practice

**❌ FORBIDDEN:**

- Implementing a method based only on its signature/name
- Assuming what a method does without reading the source
- Skipping "minor" logic or edge cases
- Literally copying TypeScript syntax patterns into C# (e.g., using `with` expressions on classes)
- Assuming C# libraries work like TypeScript libraries (e.g., FluentValidation like Zod)
- Omitting validation or error handling
- Changing business logic or validation rules

**✅ REQUIRED:**

- Read and understand the complete TypeScript method implementation
- Understand the purpose and behavior of every operation
- Implement using appropriate C# idioms and best practices
- Use correct C# library APIs (e.g., FluentValidation's actual API, not Zod's)
- Use appropriate C# patterns (e.g., direct property assignment for mutable classes, not `with` expressions)
- Preserve all functionality, validation rules, business rules, and error handling
- Ensure behavioral equivalence, not syntactic equivalence

### Example Workflow

```csharp
// Step 1: Understand TypeScript functionality
// File: core.typescript/src/application/services/ProductManagementService.ts
// Method: createProduct(dto: CreateProductDto): Promise<ProductDto>
// 
// Functionality identified:
// 1. Validate input DTO (Zod schema validation)
// 2. Check for duplicate key (repository lookup)
// 3. Create domain entity with default status
// 4. Save entity (repository save)
// 5. Return DTO (mapper transformation)

// Step 2: Implement using C# idioms
public async Task<ProductDto> CreateProductAsync(CreateProductDto dto)
{
    // Functionality 1: Validate input (using FluentValidation correctly)
    var validationResult = await _createProductValidator.ValidateAsync(dto);
    if (!validationResult.IsValid)
    {
        throw new ValidationException(
            $"Invalid product data for key '{dto.Key}': {string.Join(", ", validationResult.Errors.Select(e => e.ErrorMessage))}",
            validationResult.Errors
        );
    }
    // Note: FluentValidation doesn't have .Data property like Zod - use original dto
    
    // Functionality 2: Check for duplicate (same pattern)
    var existing = await _productRepository.FindByKeyAsync(dto.Key);
    if (existing != null)
    {
        throw new ConflictException($"Product with key '{dto.Key}' already exists");
    }
    
    // Functionality 3: Create entity (using C# object initializer, not TypeScript object literal)
    var product = new Product(new ProductProps
    {
        Key = dto.Key,
        DisplayName = dto.DisplayName,
        Description = dto.Description,
        Status = ProductStatus.Active,
        Metadata = dto.Metadata,
        CreatedAt = DateHelper.Now(),
        UpdatedAt = DateHelper.Now()
    });
    
    // Functionality 4: Save (same pattern)
    var savedProduct = await _productRepository.SaveAsync(product);
    
    // Functionality 5: Return DTO (same pattern)
    return ProductMapper.ToDto(savedProduct);
}
```

### Key C# Idioms to Use

- **Property Mutations**: Use direct assignment (`Props.Property = value`), not `with` expressions (classes don't support `with`)
- **Validation**: Use FluentValidation's actual API (`ValidateAsync`, `IsValid`, `Errors`), not Zod patterns
- **Object Creation**: Use object initializers for classes, not positional constructors mimicking TypeScript
- **Null Checks**: Use `!= null` or pattern matching, not TypeScript truthiness
- **Collections**: Use `List<T>`, `Dictionary<TKey, TValue>`, etc., not TypeScript arrays/objects
- **Async/Await**: Use `Task<T>` and `async/await`, not `Promise<T>`

### Verification Checklist for Each Method

Before considering any method complete, verify:

- [ ] Read and understood the complete TypeScript source method
- [ ] Identified all functionality (validation, repository calls, business logic, error handling)
- [ ] Implemented all functionality using appropriate C# idioms
- [ ] Used correct C# library APIs (FluentValidation, EF Core, etc.)
- [ ] Preserved all validation rules (same checks, same messages)
- [ ] Preserved all business rules (same domain logic)
- [ ] Preserved all error handling paths
- [ ] Preserved all edge cases
- [ ] Verified behavioral equivalence (not just syntactic similarity)
- [ ] Written corresponding test that verifies same behavior

### Priority Order

1. **Functional Equivalence** (MOST IMPORTANT)

   - All TypeScript functionality must be present
   - Behavior must be identical in all cases
   - All validation, business rules, and error handling must match

2. **C# Best Practices**

   - Use appropriate C# idioms and patterns
   - Use correct C# library APIs
   - Follow .NET conventions

3. **Code Quality**

   - Clean, readable, maintainable code
   - Proper use of C# 12 features

4. **Performance**

   - Optimize where possible without changing behavior
   - Use async/await patterns correctly

**If there is ANY conflict between these priorities, functional equivalence ALWAYS wins, but must be achieved using proper C# idioms.**

| Value Objects | C# Enums / Records | Enums for statuses, records for complex VOs |

| DTOs | C# Records / Classes | Records for immutable DTOs |

| Application Services | C# Classes | Same pattern |

## Key Design Decisions

### 1. UUID Generation

- **Decision**: Only `generateKey(prefix)` is needed (not `generateId()`)
- **Rationale**: TypeScript codebase only uses `generateKey()` for subscription keys in Stripe integration
- **Implementation**: Use `System.Guid.NewGuid()` or custom short ID generator

### 2. Internal Data Keys

- **Decision**: All internal primary keys use `long` (IDENTITY/BIGSERIAL)
- **Rationale**: Matches PostgreSQL BIGSERIAL and SQL Server BIGINT IDENTITY
- **Implementation**: EF Core `ValueGeneratedOnAdd()` with `HasIdentityOptions()`

### 3. Database Support

- **Decision**: Support both PostgreSQL and SQL Server
- **Rationale**: Different connection strings, may need database type flag
- **Implementation**: 
  - Constructor accepts `DatabaseType` enum (PostgreSQL, SqlServer)
  - Conditional EF Core provider initialization
  - Schema differences handled in migrations

### 4. Mapperly for Mappings

- **Decision**: Use Mapperly source generators
- **Rationale**: Compile-time generated mappers, zero runtime overhead
- **Implementation**: `[Mapper]` attribute with `[MapProperty]` for snake_case conversions

### 5. Exception Types

- **Decision**: Only implement exceptions that are actually used
- **Verified Exceptions**:
  - `DomainException` ✅ (used in entities and services)
  - `ValidationException` ✅ (used extensively)
  - `NotFoundException` ✅ (used extensively)
  - `ConflictException` ✅ (used for duplicate keys)
  - `ConfigurationException` ✅ (used in config loading)
  - `AuthException` ❌ (NOT used - remove from plan)

### 6. Repository Pattern Enforcement

- **Decision**: ALL database transactions MUST go through repositories
- **Rationale**: Maintains clean architecture, testability
- **Implementation**: 
  - Services never access `DbContext` directly
  - All queries go through repository interfaces
  - EF Core only used in repository implementations

### 7. Latest C# Constructs

- **Decision**: Use latest C# 12 features
- **Implementation**:
  - Primary constructors for services
  - Collection expressions
  - `required` properties
  - `init` accessors
  - File-scoped namespaces
  - Pattern matching enhancements
  - `System.Text.Json` for JSON (not Newtonsoft.Json)

## Public API Method Mapping

**⚠️ CRITICAL: All exterior-facing methods MUST match the TypeScript library exactly.**

The public API exposed by the `Subscrio` class and all service properties (`subscrio.products`, `subscrio.features`, etc.) must have:

- Identical method names (with C# naming conventions: `CreateProductAsync` vs `createProduct`)

- Identical return types (with C# async patterns: `Task<ProductDto>` vs `Promise<ProductDto>`)

### EF Core Provider Initialization

```csharp
public class Subscrio
{
    private readonly SubscrioDbContext _dbContext;
    
    public Subscrio(SubscrioConfig config)
    {
        var optionsBuilder = new DbContextOptionsBuilder<SubscrioDbContext>();
        
        if (config.DatabaseType == DatabaseType.PostgreSQL)
        {
            optionsBuilder.UseNpgsql(config.Database.ConnectionString);
        }
        else if (config.DatabaseType == DatabaseType.SqlServer)
        {
            optionsBuilder.UseSqlServer(config.Database.ConnectionString);
        }
        
        _dbContext = new SubscrioDbContext(optionsBuilder.Options);
        // ... initialize repositories and services
    }
}
```

### Schema Differences

- **PostgreSQL**: Uses `BIGSERIAL`, `JSONB`, schema `subscrio`
- **SQL Server**: Uses `BIGINT IDENTITY(1,1)`, `NVARCHAR(MAX)` with JSON functions, schema `subscrio`
- Handle in EF Core configurations and migrations

## Repository Pattern Enforcement

### Rule: All Database Access Through Repositories

**Services MUST NOT:**

- Access `DbContext` directly
- Use `_dbContext.Set<T>()`
- Execute raw SQL
- Use EF Core query methods directly

**Services MUST:**

- Use repository interfaces only
- Call repository methods for all data access
- Let repositories handle all database operations

**Example:**

```csharp
// ❌ WRONG - Direct DbContext access
public class ProductManagementService
{
    private readonly SubscrioDbContext _dbContext;
    
    public async Task<ProductDto> GetProductAsync(string key)
    {
        var product = await _dbContext.Products.FirstOrDefaultAsync(p => p.Key == key);
        // ...
    }
}

// ✅ CORRECT - Repository pattern
public class ProductManagementService
{
    private readonly IProductRepository _productRepository;
    
    public async Task<ProductDto> GetProductAsync(string key)
    {
        var product = await _productRepository.FindByKeyAsync(key);
        // ...
    }
}
```
