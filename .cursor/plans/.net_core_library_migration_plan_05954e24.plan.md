# .NET Core Library Migration Plan

## Overview

This plan outlines the migration of the TypeScript `@subscrio/core` library to a .NET equivalent. The .NET version will maintain the same architecture, domain model, and public API patterns while leveraging .NET-specific features and conventions.

## ⚠️ CRITICAL: Line-by-Line Functional Translation Philosophy

**THIS IS THE ABSOLUTE MOST IMPORTANT REQUIREMENT FOR THIS PROJECT.**

### Core Principle

**Every single method MUST be analyzed line-by-line from the TypeScript source code, and ALL functionality and logic MUST be exactly translated and mapped to the .NET equivalent.**

### Mandatory Process for Each Method

1. **Read the TypeScript Source First**

   - Open the exact TypeScript file containing the method
   - Read every single line of the method implementation
   - Understand every conditional, loop, validation, and operation
   - Note all edge cases, error handling, and business logic

2. **Map Every Operation**

   - Every `if` statement → C# `if` statement
   - Every `for` loop → C# equivalent loop
   - Every validation check → FluentValidation or equivalent
   - Every repository call → Repository interface call
   - Every error throw → Exception throw
   - Every data transformation → Mapper call
   - Every business rule → Domain entity method or service call

3. **Verify Complete Coverage**

   - No TypeScript logic should be omitted
   - No shortcuts or "simplifications" that change behavior
   - All error paths must be preserved
   - All validation rules must match exactly
   - All business rules must match exactly

4. **Test-Driven Verification**

   - Every TypeScript test must have a corresponding .NET test
   - Tests must verify the same behavior
   - Edge cases must be tested identically

### What This Means in Practice

**❌ FORBIDDEN:**

- Implementing a method based only on its signature/name
- Assuming what a method does without reading the source
- Skipping "minor" logic or edge cases
- "Improving" the logic during translation
- Combining or splitting operations differently
- Changing the order of operations
- Omitting validation or error handling

**✅ REQUIRED:**

- Read the TypeScript method implementation completely
- Understand every line before writing C# code
- Translate every operation, even if it seems redundant
- Preserve all conditional logic exactly
- Maintain all error handling paths
- Keep all validation rules identical
- Match all business rule enforcement

### Example Workflow

```csharp
// Step 1: Read TypeScript source
// File: core.typescript/src/application/services/ProductManagementService.ts
// Method: createProduct(dto: CreateProductDto): Promise<ProductDto>
// 
// Line 1: const validationResult = CreateProductDtoSchema.safeParse(dto);
// Line 2: if (!validationResult.success) {
// Line 3:   throw new ValidationError('Invalid product data', validationResult.error.errors);
// Line 4: }
// Line 5: const validatedDto = validationResult.data;
// Line 6: const existing = await this.productRepository.findByKey(validatedDto.key);
// Line 7: if (existing) {
// Line 8:   throw new ConflictError(`Product with key '${validatedDto.key}' already exists`);
// Line 9: }
// ... (continue line by line)

// Step 2: Translate to C# preserving every operation
public async Task<ProductDto> CreateProductAsync(CreateProductDto dto)
{
    // Line 1-4: Validation (exact same logic)
    var validator = new CreateProductDtoValidator();
    var validationResult = await validator.ValidateAsync(dto);
    if (!validationResult.IsValid)
    {
        throw new ValidationException("Invalid product data", validationResult.Errors);
    }
    
    // Line 6: Check for existing (exact same logic)
    var existing = await _productRepository.FindByKeyAsync(dto.Key);
    if (existing != null)
    {
        throw new ConflictException($"Product with key '{dto.Key}' already exists");
    }
    
    // ... (continue translating every line)
}
```

### Verification Checklist for Each Method

Before considering any method complete, verify:

- [ ] Read the complete TypeScript source method
- [ ] Identified every operation (validation, repository calls, business logic, error handling)
- [ ] Translated every operation to C# equivalent
- [ ] Preserved all conditional logic (if/else, switch, ternary)
- [ ] Preserved all loop logic (for, while, forEach)
- [ ] Preserved all error handling (try/catch, throw statements)
- [ ] Preserved all validation rules (same checks, same messages)
- [ ] Preserved all business rules (same domain logic)
- [ ] Preserved all data transformations (same mapping logic)
- [ ] Verified edge cases are handled identically
- [ ] Written corresponding test that verifies same behavior

### Priority Order

1. **Functionality Accuracy** (MOST IMPORTANT)

   - Every line of TypeScript logic must be present in C# code
   - Behavior must be identical in all cases

2. **Code Quality**

   - Use C# best practices and modern features
   - Clean, readable, maintainable code

3. **Performance**

   - Optimize where possible without changing behavior
   - Use async/await patterns correctly

4. **Style**

   - Follow C# naming conventions
   - Use appropriate .NET patterns

**If there is ANY conflict between these priorities, functionality accuracy ALWAYS wins.**

## Architecture Mapping

### TypeScript → .NET Equivalents

| TypeScript Component | .NET Equivalent | Notes |

|---------------------|-----------------|-------|

| Domain Entities | C# Classes (DDD) | Same pattern, C# properties |

| Value Objects | C# Enums / Records | Enums for statuses, records for complex VOs |

| DTOs | C# Records / Classes | Records for immutable DTOs |

| Zod Schemas | FluentValidation | Input validation |

| Drizzle ORM | Entity Framework Core | ORM with code-first migrations |

| Repository Interfaces | C# Interfaces | Same pattern |

| Application Services | C# Classes | Same pattern |

| Domain Services | C# Classes | Same pattern |

| Custom Errors | Exception Classes | Inherit from base exceptions |

| UUID Generation | `System.Guid` or custom | Only `generateKey()` is used, not `generateId()` |

| Stripe SDK | `Stripe.net` | Official Stripe .NET SDK |

| PostgreSQL | `Npgsql.EntityFrameworkCore.PostgreSQL` | EF Core provider |

| SQL Server | `Microsoft.EntityFrameworkCore.SqlServer` | EF Core provider (additional support) |

| Mappers | Mapperly | Source generators for mapping |

## Key Design Decisions

### 1. UUID Generation

- **Decision**: Only `generateKey(prefix)` is needed (not `generateId()`)
- **Rationale**: TypeScript codebase only uses `generateKey()` for subscription keys in Stripe integration
- **Implementation**: Use `System.Guid.NewGuid()` or custom short ID generator

### 2. Internal Data Keys

- **Decision**: All internal primary keys use `long` (IDENTITY/BIGSERIAL)
- **Rationale**: Matches PostgreSQL BIGSERIAL and SQL Server BIGINT IDENTITY
- **Implementation**: EF Core `ValueGeneratedOnAdd()` with `HasIdentityOptions()`

### 3. Database Support

- **Decision**: Support both PostgreSQL and SQL Server
- **Rationale**: Different connection strings, may need database type flag
- **Implementation**: 
  - Constructor accepts `DatabaseType` enum (PostgreSQL, SqlServer)
  - Conditional EF Core provider initialization
  - Schema differences handled in migrations

### 4. Mapperly for Mappings

- **Decision**: Use Mapperly source generators
- **Rationale**: Compile-time generated mappers, zero runtime overhead
- **Implementation**: `[Mapper]` attribute with `[MapProperty]` for snake_case conversions

### 5. Exception Types

- **Decision**: Only implement exceptions that are actually used
- **Verified Exceptions**:
  - `DomainException` ✅ (used in entities and services)
  - `ValidationException` ✅ (used extensively)
  - `NotFoundException` ✅ (used extensively)
  - `ConflictException` ✅ (used for duplicate keys)
  - `ConfigurationException` ✅ (used in config loading)
  - `AuthException` ❌ (NOT used - remove from plan)

### 6. Repository Pattern Enforcement

- **Decision**: ALL database transactions MUST go through repositories
- **Rationale**: Maintains clean architecture, testability
- **Implementation**: 
  - Services never access `DbContext` directly
  - All queries go through repository interfaces
  - EF Core only used in repository implementations

### 7. Latest C# Constructs

- **Decision**: Use latest C# 12 features
- **Implementation**:
  - Primary constructors for services
  - Collection expressions
  - `required` properties
  - `init` accessors
  - File-scoped namespaces
  - Pattern matching enhancements
  - `System.Text.Json` for JSON (not Newtonsoft.Json)

### EF Core Provider Initialization

```csharp
public class Subscrio
{
    private readonly SubscrioDbContext _dbContext;
    
    public Subscrio(SubscrioConfig config)
    {
        var optionsBuilder = new DbContextOptionsBuilder<SubscrioDbContext>();
        
        if (config.DatabaseType == DatabaseType.PostgreSQL)
        {
            optionsBuilder.UseNpgsql(config.Database.ConnectionString);
        }
        else if (config.DatabaseType == DatabaseType.SqlServer)
        {
            optionsBuilder.UseSqlServer(config.Database.ConnectionString);
        }
        
        _dbContext = new SubscrioDbContext(optionsBuilder.Options);
        // ... initialize repositories and services
    }
}
```

### Schema Differences

- **PostgreSQL**: Uses `BIGSERIAL`, `JSONB`, schema `subscrio`
- **SQL Server**: Uses `BIGINT IDENTITY(1,1)`, `NVARCHAR(MAX)` with JSON functions, schema `subscrio`
- Handle in EF Core configurations and migrations

## Repository Pattern Enforcement

### Rule: All Database Access Through Repositories

**Services MUST NOT:**

- Access `DbContext` directly
- Use `_dbContext.Set<T>()`
- Execute raw SQL
- Use EF Core query methods directly

**Services MUST:**

- Use repository interfaces only
- Call repository methods for all data access
- Let repositories handle all database operations

**Example:**

```csharp
// ❌ WRONG - Direct DbContext access
public class ProductManagementService
{
    private readonly SubscrioDbContext _dbContext;
    
    public async Task<ProductDto> GetProductAsync(string key)
    {
        var product = await _dbContext.Products.FirstOrDefaultAsync(p => p.Key == key);
        // ...
    }
}

// ✅ CORRECT - Repository pattern
public class ProductManagementService
{
    private readonly IProductRepository _productRepository;
    
    public async Task<ProductDto> GetProductAsync(string key)
    {
        var product = await _productRepository.FindByKeyAsync(key);
        // ...
    }
}
```
